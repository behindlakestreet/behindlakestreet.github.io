dynatrace_managed_upgrade.yml

Interactive upgrade runner for Dynatrace Managed. Presents a menu at runtime, routes to one of five upgrade types, runs pre/post checks, downloads the correct installer from Nexus, and generates a report — all in a single playbook run across six sequenced plays.

interactive menu 5 upgrade types nexus-sourced pre + post checks 6 plays
User Interaction

Interactive selection at runtime

Unlike most playbooks, this one starts with vars_prompt — it asks two questions before any task runs. Invalid input (wrong number or anything other than "yes") aborts immediately. The cluster name is auto-derived from the inventory filename.

Architecture

Six-play pipeline

The playbook is structured as six sequential plays, each targeting different hosts. The selected upgrade type is passed between plays via hostvars['localhost'].

Play 1
User Selection
hosts: localhost
vars_prompt (2 questions) Validate confirmation Validate type (1–5) Extract cluster name from inventory Map selection → upgrade type string Display selected options
Play 2
Log Preparation
hosts: controlnode
Create log directory Create/touch log file
Play 3
Execute Upgrade
hosts: primary or clusternode
(based on selection)
Record upgrade start time include_role: dynatrace_upgrade tasks_from: <type>.yml → pre_checks → download → extract → post_checks
Play 4
Generate Report
hosts: primary or clusternode
collect_upgrade_stats.yml generate_report.yml (run_once) → writes report + /tmp/dynatrace_report_filename.txt
Play 5
Update Spreadsheet
hosts: localhost
Slurp report filename from /tmp update_versions_xlsx.yml xlsx_update_mode: upgrade
Play 6
Log Finalization
hosts: controlnode
Rename log → ansible-dynatrace-<type>-<timestamp>.log Display completion banner
Cross-play variable handoff — The selection made in Play 1 is available to all subsequent plays via hostvars['localhost']['selected_upgrade'] and hostvars['localhost']['cluster_name']. The target_hosts_map dict in plays 3 and 4 translates the upgrade type to the correct host group dynamically.

File Processing

Two processing approaches

Cluster upgrades place installer files directly into the Dynatrace upgrade directory. All other types download a zip and do a two-step extraction to reach the agent binaries.

Cluster (type 3)
4 files, no extraction
1
Download .sh installer from Nexus → bin/installer/upgrade/
2
Download .sh.sig signature file
3
Download .sh.sig.tsr timestamp file
4
Download <version>.config configuration file
5
Clean up *.zip from temp dir
Dynatrace CMC detects the installer files and applies the upgrade automatically
Agent / ActiveGate / JS / Synthetic
double extraction
1
Download <type>-<version>.zip from Nexus → /opt/dynatrace-managed/temp/
2
Set permissions on temp dir (owner: dynatrace, mode: 0755)
3
Extract outer zip → temp/ (reveals agents.zip inside)
4
Ensure temp/agents/ directory exists
5
Extract agents.zip → /data/agents/
6
Set permissions on versioned agent directory
7
Clean up *.zip from temp dir
Validation

Pre and post checks

Both check files wrap every upgrade type. They can be skipped via skip_pre_checks: true or skip_post_checks: true.

Pre-checks — before download
💾
Disk space — checks available space on /opt/dynatrace-managed. Fails if less than 5 GB available.
🌐
Nexus connectivity — HTTP GET to the Nexus browse URL. Fails if unreachable (unless skip_nexus_check: true). Accepts 200, 301, 302, 403.
⚙️
Service status — records dynatrace-server systemd state before upgrade as server_status_before fact.
Post-checks — after install
⚙️
Service status — records dynatrace-server state after upgrade as server_status_after fact.
⚠️
Service running check — fails (with ignore_errors: true) if service is not active after upgrade. Surfaces the issue without blocking the report.
Both before/after status facts feed into the upgrade report.
